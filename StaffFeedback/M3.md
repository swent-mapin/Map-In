## User Experience

For this part, we wanted to see how intuitive and user-friendly the app is for real users. Beyond having good usability, did you pay attention to streamlining the interactions, is it easy to figure out, can new users start making good use of the app quickly, are the interaction flows well thought out and refined.

Overall, the navigation and architectural information are very strong. The decision to centralize most interactions within a modal window is interesting and works well in many ways, as it avoids cluttering the interface and keeps the map at the core of the application. However, this choice also hides many features, which can make the experience less intuitive for first-time users.

In particular, there is no clear indication that users need to tap the profile picture to access the profile and other features. This can feel unintuitive, as users typically expect clicking on the profile picture to lead only to the profile page. Important sections such as Friends, Memories, and Settings are hidden behind this interaction, which makes them harder to discover. Overall, having a visible menu and a dedicated profile button (for example, in the top-right corner) might have improved discoverability.

That said, once users become familiar with the application, navigation is clear and efficient. Visual consistency is excellent, with only minor padding issues, notably around badges. Both dark and light themes work well, and the application is visually polished. The use of icons for events on the map is particularly effective and enhances the overall aesthetic. The messaging system works well and is visually appealing, contributing to a consistent user experience.

Regarding guidance and onboarding, the initial account creation flow, especially password creation, is very well done and clear. However, once users enter the application, there is very little explanation of how to use it. Adding contextual hints or a short onboarding flow on first use would have improved the experience. That said, because the overall structure is well designed, users are still able to understand the app relatively quickly.

From a UX perspective, the application is well polished and offers strong, original features. Centering the experience around the map is a good design choice, and key elements such as the search bar, map, profile access, and event creation are quickly accessible. The search bar in particular is very effective for quickly finding people or events.

One minor area for improvement is the event detail view. While it is possible to see who created an event, this information could be made more visible, for example, by displaying the creator‚Äôs profile picture or a more prominent button. Overall, these are small details, and the application is very well executed.

For this part, you received 5.4/6.

We expect the codebase to be polished, well documented, follow consistent conventions, be modular, and allow for easy modifications.
You should be able to employ advanced techniques by now, such as asynchronous functions (flows, coroutines), good resource management, and automated dependency injection (e.g., with Hilt).

We also assessed your overall app architecture and design, looking in particular at aspects surrounding robustness and scalability.
We looked at both the codebase and the documentation of the app (Wiki and architecture diagram), we expect your design to demonstrate thoughtful consideration for performance, maintainability, and future growth.

The project has good architecture and modularity. Overall, the packaging is clear, responsibilities are well separated. It would be better to place the navigation package under ui, as navigation belongs to the presentation layer and grouping it there would improve consistency. Under the model layer, the ai package contains many files, which makes it harder to understand how the different components relate to each other. Several files also begin with the prefix Ai, which is redundant given that they are already inside the ai package. The package could benefit from clearer internal structure, for example, separating speech-to-text and text-to-speech services into their own subpackage and grouping related components more explicitly.

There are only two high-severity Sonar issues left, and it‚Äôs clear that significant progress has already been made in reducing them, which is great to see. Since these remaining issues involve high cognitive complexity, addressing them would still be beneficial for long-term maintainability and would make those parts of the code easier to work with in the future.
